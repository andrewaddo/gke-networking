# L4 Load Balancer with TLS Passthrough Sample

This sample demonstrates how to configure a GKE cluster with a Layer 4 (L4) Internal Load Balancer where **TLS is passed through** to the application pods (the load balancer does not decrypt the traffic; the Pods terminate the TLS connection).

```
[ Client ]
    │  (Encrypted HTTPS traffic - Port 443)
    ▼
[ L4 Internal Passthrough Load Balancer ] (No TLS Termination)
    │  (Encrypted HTTPS traffic - Port 443)
    ▼
[ GKE Node ]
    │  (Encrypted HTTPS traffic - Port 443)
    ▼
[ Pod (Nginx) ] (TLS Terminated Here using self-signed certs)
```

## Setup Files
*   `nginx.conf`: Nginx configuration mapping port 443 to SSL and pointing to certs.
*   `nginx-deployment.yaml`: Deployment creating Nginx pods, mounting the config and SSL certs.
*   `nginx-service.yaml`: Service of `type: LoadBalancer` (configured as Internal) exposing port 443.
*   `setup.sh`: Script to automate certificate generation, Secret/ConfigMap creation, and deployment.

## Deployment Steps

1.  Run the setup script to generate self-signed certificates and deploy the resources:
    ```bash
    ./setup.sh
    ```

2.  Monitor the service until it gets an External IP address:
    ```bash
    kubectl get svc nginx-tls-service -w
    ```

## Verification Steps

Once you have the internal IP (let's say it is `10.128.0.x`), verify that TLS passthrough is working.

> [!IMPORTANT]
> Because this is an Internal Load Balancer, you must run these verification steps from a VM or Pod **within the same VPC network** (or from within the cluster).

1.  **Test Connectivity:**

    **Option A: From a temporary Pod in the cluster (Easiest)**
    If you have `kubectl` access, you can spin up a temporary Pod to run the test:
    ```bash
    kubectl run curl-test --rm -i --image=curlimages/curl --restart=Never -- -k -v https://<INTERNAL_IP>
    ```

    **Option B: From a VM in the same VPC**
    If you are running the test from a VM residing in the same VPC network:
    ```bash
    curl -k -v https://<INTERNAL_IP>
    ```
    *(We use `-v` to inspect the TLS handshake and verify the certificate, and `-k` because the certificate is self-signed).*

2.  **Verify Certificate Details:**
    You can inspect the certificate returned by the server to confirm it is the one generated by the script:
    ```bash
    openssl s_client -connect <INTERNAL_IP>:443 -showcerts
    ```
    In the output, look for the Subject line. It should match the subject we configured in `setup.sh`:
    ```
    depth=0 CN = nginx-tls-sample, O = GKE-Networking-Test
    ```

3.  **Confirm Termination is at the Pod:**
    Check the logs of the Nginx pods. You should see the TLS connection being processed directly by the pod:
    ```bash
    kubectl logs -l app=nginx-tls
    ```
